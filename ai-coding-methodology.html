一、为什么需要 AI Coding 工程化？

1.1 AI 编码的四大痛点

痛点表现后果
上下文碎片化AI 每次对话只能看到有限文件代码与已有架构冲突
缺乏记忆上一轮做了什么、设计约束是什么，AI 全忘了重复沟通、风格不一致
无质量兜底没有 checklist，AI 改了 A 忘了 B暗色模式遗漏、文档过时
版本失控没有分支策略，直接在 master 上推无法回退、无法并行开发

1.2 我们的解决方案

Spec-Driven Development

让 AI 在「文档围栏」内编码

文档是 AI 的上下文、约束和验收标准

核心思路
通过文档驱动开发，让 AI 编码从「随性创作」变成「精确执行」。
文档就是 AI 的记忆、约束和检查清单。

工作流程：

📄 文档 → 驱动 → 🤖 AI → 产出 → 💻 代码
↑_______同步更新_______↓

二、六层文档体系 — AI 的项目大脑

我们构建了 6 份核心文档 的项目知识库，作为 AI 编码的持久化上下文：

💡 项目文档结构
项目根目录
├── architecture.md          # [架构] 架构蓝图
├── data-model.md            # [数据] 数据模型
├── coding-conventions.md   # [规范] 开发规范
├── ui-design-system.md     # [设计] 设计系统
├── component-library.md    # [组件] 组件库
└── service-api.md           # [接口] 服务层 API
文档AI 读取时机AI 获得的能力
architecture.md任何开发开始前知道文件放哪里、模块依赖关系
data-model.md涉及数据操作时正确构造查询、不漏字段
coding-conventions.md写代码时遵循命名规范、使用正确模式
ui-design-system.md涉及 UI 时使用 CSS 变量而非硬编码
component-library.md使用/修改组件时正确传 props、触发 events
service-api.md调用服务时正确的方法签名和参数
铁律：每次 feature 完成后，必须同步更新受影响的文档。
确保 AI 在下一次对话中读到的文档永远是最新的。
实战效果
有了这 6 份文档，AI 就像拥有了项目的「完整记忆」。
不用每次重新解释项目结构，直接开始干活！

三、Spec 三件套 — Feature 开发的标准蓝图

每个功能迭代在 specs/ 目录下创建标准化的三份文档：

💡 Spec 三件套结构
specs/<feature-name>/
├── requirements.md    # [需求] 需求文档 — 做什么
├── design.md           # [设计] 设计文档 — 怎么做
└── tasks.md             # [任务] 任务列表 — 做到哪
📄 requirements.md — 定义「做什么」

核心内容：用户故事 + 验收标准

采用 When...shall... 格式，让 AI 能够精确对齐每一行代码和对应的需求。

🎨 design.md — 定义「怎么实现」

核心内容：CSS Diff 表 + WXML 改造策略 + JS 交互变更

CSS Diff 表是 AI 最重要的参照物。AI 逐行执行 diff，不需要猜测。

✅ tasks.md — 定义「执行顺序」

核心内容：任务清单 + 完成状态

每个任务 T-x.x 完成后立即打勾。tasks.md 就是项目的实时进度看板。

为什么需要三件套？
❌ 没有 requirements：AI 不知道要做什么
❌ 没有 design：AI 需要猜测如何实现
❌ 没有 tasks：AI 不知道执行顺序和进度
✅ 三件套齐全：AI 清清楚楚、零返工

四、Git Flow — 版本控制纪律

AI 编码也需要严格的版本控制，否则出了问题连回退的机会都没有。

4.1 分支策略

分支命名规则
master—禁止直接 push，仅 PR 合并
develop—集成分支，feature 合入此处
feature/<spec-name>与 specs/ 目录名一致从 develop 拉，合回 develop
release/v<x.y.z>语义化版本号从 develop 拉，合并 master + develop

4.2 Git Flow 工作流程

📌 典型开发流程示例
步骤 操作 说明
1 从 develop 拉分支 创建 feature/ui-redesign-v4 分支
2 在 feature 分支开发 完成 32 个任务，提交 23 个 commits
3 合回 develop 功能测试通过后合并到 develop 分支
4 创建 release 分支 准备发布 v4.0.0
5 合并到 master 打 tag v4.0.0，正式发布
🔄 分支流转示意
develop → feature/xxx → develop → master
集成分支 功能开发 测试通过 发布上线
Commit 铁律：
✓ 遵循 Conventional Commits
✓ 一个 Task 至少一个 commit
✓ 每个 commit 可独立编译运行
✓ 关联 T-x.x 和 FR-x 编号
为什么重要？
有了 Git Flow，你可以：
✓ 随时回退到任意版本
✓ 多个 feature 并行开发
✓ 清晰的版本演进轨迹

五、三阶段开发工作流

完整的 feature 开发流程分为三个阶段，确保质量闭环：

5.1 开发前：给 AI 装载上下文

  1. 阅读 6 份核心文档 → AI 理解项目全貌
  2. 编写 Spec 三件套 → AI 知道「这次做什么」
  3. 10 轮交叉 Review → 和 AI 一起检查遗漏（v4.0 发现了 30 处遗漏）
  4. 创建执行计划 → 按天排期，每天有验收节点

5.2 开发中：AI 在围栏内执行

关键指令模式：

💡 关键指令模式
开始 Day X，确保实施和设计方案对齐 @specs/<feature>/design.md

AI 执行循环：

📖 读 design.md → 📝 读代码 → ✏️ 按 Diff 表修改 → 💾 commit → ☑️ 更新 checkbox → 📤 push

5.3 开发后：闭环文档同步

检查 6 份文档：

  • 新增了组件？→ 更新 component-library.md
  • 改了目录结构？→ 更新 architecture.md
  • 新增了 CSS 变量？→ 更新 ui-design-system.md
  • 新增了服务方法？→ 更新 service-api.md
关键要点
开发前：AI 需要充分的上下文
开发中：AI 在文档围栏内执行
开发后：必须同步更新文档
形成闭环，确保下一次对话 AI 读到的是最新状态

6.4 关键决策案例

发现问题

「今日概览」点击开弹窗 = 「快捷记录」点击开弹窗
80% 功能重叠，造成用户困惑

头脑风暴 5 个方案：

  • 选定 A: 概览回归纯展示（跳转记录页）
  • B: 去掉快捷记录横栏
  • C: 概览点击展开详情浮层
  • D: 合并为智能卡片
  • E: 概览保留 + 快捷改 FAB

选择理由：职责分离 — 概览「看数据」，快捷「做记录」，心智模型清晰，改动最小。

决策过程
这就是 Spec-Driven Development 的价值：
在设计阶段就发现潜在问题，通过头脑风暴找到最优解，
避免开发到一半才发现架构不合理。

七、AI 协作要点 — 如何让 AI 听懂你的话

7.1 指令最佳实践

✅ 好的指令示例
指令 AI 理解
开始 Day 4，确保实施和设计方案对齐 @specs/ui-redesign-v4/design.md 读 design.md → 找 Day 4 任务 → 对齐实施
再 review 10 次 design.md，确保没有遗漏 逐条交叉比对 → 列出遗漏清单 → 更新文档
❌ 差的指令示例
指令 AI 困惑
帮我改一下首页样式 改哪些？改成什么样？和什么规格对齐？

7.2 上下文管理策略

策略做法效果
持久化记忆项目信息存入 AI memoryAI 记住架构、文档体系、开发工作流
文件引用指令中用 @fileAI 自动加载相关文档
渐进式加载先读框架 → 再读章节 → 再读代码避免上下文过载
并行读取一次读取多个文件减少交互轮次

7.3 质量保障机制

机制实施方式保障内容
Spec Review生成 design.md 后进行 10 轮交叉 review发现遗漏、确保完整性
逐日验收plan.md 每天有验收节点表确保每日进度可量化
Commit 粒度每个 Task 一个 commit，可独立编译可追溯、可回退
暗色矩阵28 项逐一检查确保暗色模式无遗漏
文档同步开发后必须更新 6 份核心文档确保文档时效性
核心原则
给 AI 的指令要：具体、可执行、有参照物
❌ "帮我改一下" → AI 无所适从
✅ "按 design.md 第 3 节修改首页" → AI 精确执行
