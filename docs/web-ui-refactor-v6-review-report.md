## 3. 响应式适配检查（⚠️ 部分遗漏）

### 3.1 当前覆盖的断点

| 断点 | 宽度 | v6-plan.md 覆盖情况 |
|------|------|----------------|
| 移动端 | < 640px | ✅ 明确（底部 TabBar、弹窗底部弹出） |
| 平板端 | 640-1024px | ✅ 明确（侧边栏 64px 仅图标） |
| 桌面端 | > 1024px | ✅ 明确（侧边栏 220px 图标+文字） |

### 3.2 遗漏点（⚠️ 需补充）

| 问题 | 影响 | 建议 |
|------|------|------|
| **大屏手机（428px，如 iPhone Pro Max）** | v6-plan.md 只写了 <640px 移动端，但 428px 在竖屏时可能挤满 640px 容器，导致部分布局过挤 | 补充「移动端大屏（428px）」测试点：检查卡片列数、字体是否过小 |
| **平板竖屏（768×1024）** | 侧边栏 64px + 内容区 704px，空间充足但 v6-plan 未利用 | 建议：平板竖屏时卡片网格从 1 列改为 2 列（当前仅桌面端 2 列） |
| **超宽屏（>1920px）** | 内容区 max-width: 1200px 居中，两侧留白过多 | 建议：超宽屏时 max-width 提升到 1400px 或 1600px |

### 3.3 动效在移动端的性能风险

| 动效 | 移动端风险 | 建议 |
|------|---------|------|
| `card-stagger` 交错动画 | 列表过长时（>20 条）可能导致首屏卡顿 | 限制 stagger 最大延迟（例如最多 0.3s，后续卡片无延迟） |
| `numberRoll` 数字滚动 | 4 个数字同时动画，低端机可能掉帧 | 使用 `transform: translateY()` 而非 `top`（GPU 加速） |
| 玻璃态 `backdrop-filter` | 低端 Android 不支持或卡顿 | 提供 fallback：不支持 `backdrop-filter` 时使用纯色背景 |

### 3.4 响应式适配结论

**建议补充**：
1. 在 §3（设计系统升级方案）中新增「响应式动效策略」：
   - 移动端：减少 stagger 延迟（最大 0.2s）
   - 平板/桌面：完整 stagger（最大 0.4s）
2. 在 §8（风险与应对）中补充「移动端大屏适配」风险项

## 4. 性能风险检查（⚠️ 需补救）

### 4.1 可能引起性能问题的改动

| 改动 | 性能风险 | 影响设备 | 建议方案 |
|------|---------|---------|---------|
| 渐变背景（§3.1 `--gradient-*`） | GPU 渲染压力（渐变比纯色费 GPU） | 低端手机、老款 Mac | 限制渐变使用范围（仅关键区域：登录页、报告封面）；列表项禁用渐变 |
| 玻璃态（§3.1 `--glass-*`） | `backdrop-filter: blur(12px)` 极费 GPU | 低端 Android、iPhone 6s 及以下 | 检测支持性：`CSS.supports('backdrop-filter: blur(12px)')`，不支持时 fallback 纯色 |
| 增强阴影（§3.1 `--shadow-card-elevated` 等） | 阴影越多，合成层数越多 | 低端设备 | 移动端禁用 `shadow-float`（仅桌面端使用） |
| 动画数量（§3.3 动效规范） | 同时运行多个 animation 会占满主线程 | 所有设备 | `prefers-reduced-motion` 必须实现（见 §5） |

### 4.2 `prefers-reduced-motion` 具体实现方案（⚠️ 缺失）

**当前状态**：v6-plan.md §8 风险表提到 `prefers-reduced-motion`，但**无具体实现方案**。

**必须补充的 CSS**：

```css
/* 放到 globals.css 顶部 */
@media (prefers-reduced-motion: reduce) {
  *,
  *::before,
  *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }

  /* 保留 opacity 变化（不引起眩晕）*/
  .fade-in {
    animation: none;
    opacity: 1;
  }
}
```

**实施步骤**：
1. 在 Phase 0（设计系统升级）中新增任务：
   - [ ] 在 `globals.css` 中添加 `prefers-reduced-motion` 媒体查询
2. 在 Phase 4（测试与文档）中新增测试项：
   - [ ] Safari → 设置 → 辅助功能 → 动效 → 减弱动效，验证所有动画禁用

### 4.3 性能风险结论

**建议**：
1. 在 §8（风险与应对）中补充 2 个新风险项：
   | 风险 | 应对措施 |
   |------|---------|
   | `backdrop-filter` 在低端设备不支持或卡顿 | 使用 JS 检测支持性，不支持时 fallback 纯色背景 |
   | 动画数量过多导致主线程阻塞 | 实现 `prefers-reduced-motion` 媒体查询（见 §5） |

2. 在 Phase 0 任务清单中补充 `prefers-reduced-motion` 实现任务

## 7. 与小程序端设计一致性检查（⚠️ 有冲突）

### 7.1 设计语言对比

| 设计元素 | 小程序端（参考 `ui-design-system.md`） | Web 端 v6 方案（§3） | 一致性 |
|-----------|--------------------------|---------------------|---------|
| 主色 | `#D4B896` | `#D4B896` | ✅ 一致 |
| 功能色 | feeding/sleep/diaper/temperature/growth | 同左 | ✅ 一致 |
| 卡片样式 | 扁平化（v4.3），无阴影，border 区分层级 | 新增阴影（`--shadow-card-elevated` 等）、玻璃态 | ⚠️ **冲突**（Web 有阴影/玻璃态，小程序无） |
| 渐变背景 | **无**（v4.3 已去除所有渐变） | 新增 6 个渐变 Token（§3.1） | ⚠️ **冲突**（Web 有渐变，小程序无） |
| 动效 | `fadeInUp`、`slideUp`、`buttonPress` 等 | 新增 `numberRoll`、`breatheGlow`、`card-stagger` | ⚠️ **分化**（Web 动效更丰富） |

### 7.2 潜在问题

| 问题 | 影响 | 建议 |
|------|------|------|
| **设计语言分化** | 用户同时用小程序 + Web 时，体验不一致 | 建议：1) 小程序端跟进渐变/玻璃态（如果性能允许）；或 2) Web 端去除渐变/玻璃态，保持扁平化 |
| **品牌感提升方向不同** | 小程序靠「卡片扁平化 + 图标圆底」，Web 靠「渐变 + 玻璃态」 | 建议：统一品牌感提升方向，选择一种设计语言作为主线 |
| **动效丰富度不同** | Web 动效远丰富于小程序 | 建议：小程序端也增加 `numberRoll` 等动效（如果性能允许） |

### 7.3 与小程序端设计一致性结论

**关键决策建议**：

> **选项 A（推荐）**：Web 端 v6 方案**回退**渐变和玻璃态，保持扁平化设计，与小程序端一致。
> - 优点：设计语言统一，用户跨端体验一致
> - 缺点：品牌感提升不如预期
>
> **选项 B**：小程序端**跟进** Web 端的渐变和玻璃态（如果性能允许）。
> - 优点：Web 端品牌感提升最大化
> - 缺点：小程序端性能风险（尤其是低端机）
>
> **选项 C**：两端**故意分化**，Web 端更「现代」，小程序端更「轻量」。
> - 优点：各自发挥平台优势
> - 缺点：用户体验不一致，设计维护成本高

**当前建议**：在 v6-plan.md 中补充「设计一致性说明」，明确说明为何 Web 端引入了小程序端没有的渐变/玻璃态，以及未来小程序端是否跟进。
