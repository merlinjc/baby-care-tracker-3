# AI Coding 工程化方法论：Spec-Driven Development 实践

> **作者**: Baby Care Tracker 团队 | **日期**: 2026-04-13 | **版本**: v1.0
> **项目**: Baby Care Tracker — 微信小程序，18 个页面 + 16 个组件，CloudBase 云开发

---

## 一、为什么需要 AI Coding 工程化？

### 1.1 AI 编码的核心痛点

| 痛点 | 表现 | 后果 |
|------|------|------|
| **上下文碎片化** | AI 每次对话只能看到有限文件，不了解项目全貌 | 生成的代码与已有架构冲突 |
| **缺乏记忆** | 上一轮做了什么、为什么做、设计约束是什么，AI 全忘了 | 重复沟通、风格不一致 |
| **无质量兜底** | 没有 checklist，AI 改了 A 忘了 B | 暗色模式遗漏、文档过时、命名混乱 |
| **版本失控** | 没有分支策略，所有改动在 master 上直接推 | 无法回退、无法并行开发 |

### 1.2 我们的答案：Spec-Driven Development

```
       ┌──────────────────────────────────────────┐
       │          Spec-Driven Development          │
       │                                            │
       │   文档 ← 驱动 → AI ← 产出 → 代码           │
       │     ↑                          │           │
       │     └──── 同步更新 ←───────────┘           │
       └──────────────────────────────────────────┘
```

**核心理念**：让 AI 在「文档围栏」内编码 —— 文档是 AI 的上下文、约束和验收标准。

## 二、六层文档体系

我们构建了一个 **6 份核心文档** 的项目知识库，作为 AI 编码的持久化上下文：

```
项目根目录
├── architecture.md          # 🏗️ 架构蓝图（目录树/分层架构/分包策略/初始化流程）
├── data-model.md            # 💾 数据模型（6 个集合字段定义/缓存策略/原子操作）
├── coding-conventions.md    # 📏 开发规范（命名约定/4 种代码模式/错误处理/性能规范）
├── ui-design-system.md      # 🎨 设计系统（美拉德色板/8rpx 网格/动画定义）
├── component-library.md     # 🧩 组件库（16 个组件的属性/事件/功能说明）
└── service-api.md           # ⚡ 服务层 API（11 个服务的方法签名/返回结构）
```

### 每份文档的 AI 作用

| 文档 | AI 读取时机 | AI 获得的能力 |
|------|------------|--------------|
| `architecture.md` | 任何开发开始前 | 知道文件放哪里、模块依赖关系 |
| `data-model.md` | 涉及数据操作时 | 正确构造查询、不漏字段 |
| `coding-conventions.md` | 写代码时 | 遵循命名规范、使用正确模式 |
| `ui-design-system.md` | 涉及 UI 时 | 使用 CSS 变量而非硬编码、引用正确动画 |
| `component-library.md` | 使用/修改组件时 | 正确传 props、触发 events |
| `service-api.md` | 调用服务时 | 正确的方法签名和参数 |

### 文档与代码的同步协议

**铁律**：每次 feature 完成后，必须同步更新受影响的文档。

```
代码变更 → 检查 6 份文档 → 更新受影响的文档 → 提交
```

这确保了 AI 在下一次对话中读到的文档永远是最新的。

## 三、Spec 三件套 — Feature 开发的蓝图

每个功能迭代在 `specs/` 目录下创建标准化的三份文档：

```
specs/<feature-name>/
├── requirements.md    # 📋 需求文档 — 做什么
├── design.md          # 📐 设计文档 — 怎么做
└── tasks.md           # ✅ 任务列表 — 做到哪
```

### 3.1 requirements.md — 需求文档

**定义「做什么」和「做到什么程度」**

核心结构：
```markdown
## 概述（100 字以内，一句话说清楚范围）

## 用户角色 & 场景表（谁在什么场景下用）

## FR-1 ~ FR-N：功能需求
  - 用户故事：作为<角色>，我想要<目标>，以便<价值>
  - 验收标准：When <条件>, the system shall <行为>
  - 技术说明：关键实现提示

## NFR-1 ~ NFR-N：非功能需求（性能/兼容性）

## 变更影响范围表（文件 × 改动类型 × 涉及 FR）
```

**关键实践**：验收标准使用 `When...shall...` 格式，让 AI 能够**精确对齐**每一行代码和对应的需求。

### 3.2 design.md — 设计文档

**定义「怎么实现」**

核心结构：
```markdown
## 架构概览（Mermaid 流程图 / ASCII 框图）

## CSS Diff 表（v3.x → v4.0 逐属性对比）

## WXML 结构改造策略（删什么/改什么/加什么）

## JS 交互变更（方法签名变化/新增方法）

## 文件变更清单（每个文件的改动类型和主要变更）

## 关键设计决策（方案 A vs 方案 B 及选择理由）
```

**关键实践**：design.md 中的 **CSS Diff 表** 是 AI 最重要的参照物。AI 逐行执行 diff，不需要猜测。

### 3.3 tasks.md — 任务列表

**定义「执行顺序」和「当前进度」**

核心结构：
```markdown
## 实施概览（总工时/里程碑 M1~MN）

### 阶段一（M1）
- [ ] T-1.1 <任务标题>
  - <具体步骤>
  - 验收：<如何验证>
  - _依赖：T-x.x | 涉及：FR-x_

## 任务依赖关系（ASCII DAG 图）

## 风险说明表
```

**关键实践**：每个任务完成后立即 `- [ ]` → `- [x] ✅`，tasks.md 就是项目的**实时进度看板**。

## 四、Git Flow 工作流 — 版本控制纪律

### 4.1 分支模型

```
master:    ●──────────────────●──────● ...    ← 生产版本 + 标签
                               ↑
develop:   ●───●───●───●───●───●───● ...    ← 集成开发
               ↑       ↑       ↑
feature:  ●────●  ●────●  ●────●            ← 每个 spec 一个分支
```

| 分支 | 命名 | 规则 |
|------|------|------|
| `master` | — | 禁止直接 push，仅 PR 合并 |
| `develop` | — | 集成分支，feature 合入此处 |
| `feature/<spec-name>` | 与 specs/ 目录名一致 | 从 develop 拉，合回 develop |
| `release/v<x.y.z>` | 语义化版本号 | 从 develop 拉，合并 master + develop |

### 4.2 Commit 规范

遵循 Conventional Commits，每个 commit 关联 Task 编号和 FR 编号：

```
feat(home): 去掉 baby-card，合并多宝切换到问候区 + 状态胶囊改造 FR-2/3
style(popup,form): 弹窗统一升级 + 表单按钮高度统一 FR-9
fix(home): onStatTap 改回跳转记录页带筛选
docs: v4.0 文档同步 — architecture/ui-design-system/component-library/README
```

**铁律**：一个 Task 至少一个 commit，每个 commit 可独立编译运行。

## 五、三阶段开发工作流 — 完整执行循环

```
┌─────────────────────────────────────────────────────┐
│                  Feature 迭代循环                      │
│                                                       │
│   开发前              开发中              开发后         │
│   ┌─────┐          ┌─────┐          ┌─────┐         │
│   │阅读6份│  ──→    │按Task│  ──→    │同步6份│         │
│   │核心文档│         │逐个   │         │核心文档│         │
│   │编写Spec│         │commit │         │更新版本│         │
│   │三件套  │         │更新   │         │创建PR │         │
│   │拉分支  │         │checkbox│        │清理分支│         │
│   └─────┘          └─────┘          └─────┘         │
│                                                       │
│   产出物：            产出物：            产出物：         │
│   requirements.md    N 个 commit       文档一致          │
│   design.md          tasks.md ✅       PR merged        │
│   tasks.md                             feature 删除     │
│   feature 分支                                          │
└─────────────────────────────────────────────────────┘
```

### 5.1 开发前：给 AI 装载上下文

1. **阅读 6 份核心文档** → AI 理解项目全貌
2. **编写 Spec 三件套** → AI 知道「这次要做什么」
3. **10 轮交叉 Review** → 和 AI 一起检查 spec 的遗漏（我们在 v4.0 中发现了 30 处遗漏）
4. **创建执行计划 plan.md** → 按天排期，每天有验收节点

### 5.2 开发中：AI 在围栏内执行

与 AI 协作的关键指令模式：

```
开始 Day X，确保实施和设计方案对齐 @specs/<feature>/design.md
```

AI 的执行循环：
1. 读取 design.md 对应章节
2. 读取当前代码文件
3. 按 CSS Diff / WXML 改造策略逐项修改
4. 提交（commit message 关联 T-x.x 和 FR-x）
5. 更新 tasks.md checkbox
6. 推送

### 5.3 开发后：闭环文档同步

```bash
# 检查 6 份文档是否需要更新
architecture.md      → 新增了组件？改了目录结构？
data-model.md        → 改了数据模型？
coding-conventions.md → 新增了代码模式？
ui-design-system.md  → 新增了 CSS 变量/动画？
component-library.md → 新增了组件？
service-api.md       → 新增了服务方法？
```

## 六、实战案例：UI Redesign v4.0

### 6.1 项目概况

| 指标 | 数据 |
|------|------|
| 影响范围 | 18 页面 + 16 组件 |
| 总任务数 | 32 个（30 原计划 + 2 补充） |
| 总 Commit | 23 个 |
| 开发周期 | 8 天（Day 1 ~ Day 8） |
| 新增组件 | 1 个（focus-card） |
| 新增 CSS 变量 | 6 个 + 暗色覆盖 4 个 |
| 新增动画 | 3 个 |
| 文档同步 | 4 份核心文档更新 |

### 6.2 Spec 文档层次

```
specs/ui-redesign-v4/
├── design-spec.md      # 35KB — 设计师产出的原始设计总纲
├── detailed-spec.md    # 22KB — 18 页面 + 15 组件逐项细化规格
├── requirements.md     # 标准 Spec — 14 FR + 4 NFR
├── design.md           # v2.0 — 10 轮 review 后的实施设计
├── tasks.md            # v2.0 — 6 阶段 32 个任务
└── plan.md             # 8 天每日执行计划
```

**亮点**：设计师交付 `design-spec.md` + `detailed-spec.md`（共 57KB），然后通过 10 轮交叉 review 生成覆盖全面的实施 `design.md`。这个 review 过程发现了 **30 处遗漏**，包括 9 个弹窗的内部样式升级、12 个分包页面的视觉统一。

### 6.3 8 天执行时间线

| Day | 阶段 | Commits | 核心产出 |
|-----|------|---------|----------|
| 1 | M1 基础层 | 3 | CSS 变量 + 动画 + 图标 + popup/form 样式 |
| 2 | M2 前半 | 1 | 问候区精简 + 状态胶囊 |
| 3 | M2 后半 | 1 | 摘要进度条 + 快捷入口 + 待办竖向 + AI 一行式 + 轴线 |
| 4 | M3 | 3 | 记录页吸顶 + focus-card + 发现页重构 + 弹窗底部化 |
| 5 | M4 | 3 | 我的页重构 + 页头清理(3页) + 弹窗关闭按钮(13处) |
| 6 | M5 前半 | 3 | feeding 常用量 + sleep/diaper/temp 升级 + growth 对比 |
| 7 | M5 后半 | 2 | packageGrowth/Social/auth/baby-create 视觉升级 |
| 8 | M6 | 3 | 细节打磨 + 暗色 QA(28项) + 文档同步(4份) |
| 补充 | 交互优化 | 1 | 概览点击行为回归（头脑风暴后决策） |

### 6.4 关键决策示例：交互功能重复

**发现问题**：「今日概览」点击开弹窗 = 「快捷记录」点击开弹窗，80% 功能重叠。

**头脑风暴 5 个方案**：
- A: 概览回归纯展示（跳转记录页）✅ **选定**
- B: 去掉快捷记录横栏
- C: 概览点击展开详情浮层
- D: 合并为智能卡片
- E: 概览保留 + 快捷改 FAB

**选择理由**：职责分离 —— 概览「看数据」，快捷「做记录」，心智模型清晰，改动最小。

## 七、AI 协作要点

### 7.1 给 AI 下指令的最佳实践

```
✅ 好的指令：
"开始 Day 4，确保实施和设计方案对齐 @specs/ui-redesign-v4/design.md"
→ AI 知道：读 design.md → 找 Day 4 任务 → 对齐实施

✅ 好的指令：
"再 review 10 次 design.md，确保没有遗漏"
→ AI 知道：逐条交叉比对 → 列出遗漏清单 → 更新文档

❌ 差的指令：
"帮我改一下首页样式"
→ AI 不知道：改哪些？改成什么样？和什么规格对齐？
```

### 7.2 AI 上下文管理

| 策略 | 做法 |
|------|------|
| **持久化记忆** | 项目信息存入 AI 的 memory（架构、文档体系、开发工作流） |
| **文件引用** | 指令中用 `@file` 引用具体文档，AI 自动加载 |
| **渐进式加载** | 先读 design.md 框架 → 再读具体章节 → 再读代码文件 |
| **并行读取** | 一次读取多个文件，减少交互轮次 |

### 7.3 质量保障机制

| 机制 | 实施方式 |
|------|----------|
| **Spec Review** | 生成 design.md 后进行 10 轮交叉 review |
| **逐日验收** | plan.md 每天有验收节点表 |
| **Commit 粒度** | 每个 Task 一个 commit，可独立编译 |
| **暗色矩阵** | 28 项逐一检查（detailed-spec §7） |
| **文档同步** | 开发后必须更新 6 份核心文档 |

## 九、核心收益

### 9.1 效率提升

| 维度 | 无 Spec | 有 Spec | 提升 |
|------|---------|---------|------|
| AI 返工次数 | 频繁（猜错了改） | 极少（有 Diff 表对齐） | **↓ 80%** |
| 沟通轮次 | 每步确认 | 一句指令执行一天 | **↓ 70%** |
| 遗漏率 | 高（改 A 忘 B） | 低（10 轮 review + checklist） | **↓ 90%** |

### 9.2 质量保障

| 维度 | 效果 |
|------|------|
| **代码一致性** | CSS 变量引用率 100%，无硬编码颜色 |
| **文档时效性** | 6 份核心文档始终与代码同步 |
| **版本可追溯** | 23 个 commit，每个关联 Task 和 FR 编号 |
| **暗色模式** | 28 项矩阵逐一验证，全部通过 |

### 9.3 可复制性

整套方法论不依赖特定项目，核心组件可直接复用：

- `specs/workflow/git-flow.md` → 任何项目的版本管理
- `specs/workflow/development-workflow.md` → 任何项目的 AI 编码流程
- Spec 三件套模板 → 任何 feature 的标准化交付物
- 6 份核心文档体系 → 任何项目的 AI 上下文基座

## 十、总结

```
                    Spec-Driven Development
                    
     ┌─────────────────────────────────────────┐
     │                                           │
     │   📋 Spec 三件套    →  AI 知道做什么       │
     │   📚 6 份核心文档   →  AI 理解项目全貌     │
     │   🌊 Git Flow       →  版本可控可回退      │
     │   🔄 三阶段工作流   →  质量有保障有闭环     │
     │   📊 10 轮 Review   →  遗漏率降到最低      │
     │                                           │
     └─────────────────────────────────────────┘
     
     AI 不是取代人，而是在「文档围栏」内高效执行。
     人负责决策和设计，AI 负责实现和验证。
     文档是两者之间的桥梁。
```
