# 需求文档 - 家庭协作功能增强

## 概述

基于现有家庭协作功能的深入分析，本需求文档针对5个核心场景进行功能增强和优化，提升多成员协作体验和权限管理能力。

**当前功能状态：**
- ✅ 家庭创建与解散
- ✅ 邀请码加入家庭（6位字母数字，7天有效期）
- ✅ 基础角色系统（admin/editor）
- ✅ 成员列表展示
- ✅ 退出家庭
- ⚠️ 成员移除功能（服务层 `family.js` L256 已实现，前端未集成）
- ⚠️ 权限变更功能（服务层 `family.js` L307 已实现，前端未集成）
- ❌ 角色权限细分（viewer角色未启用）
- ❌ 前端权限控制（UI 未根据角色限制操作）
- ❌ 操作者身份标识（记录无创建者展示）

**家庭管理页路径：** `pages/family/family`

---

## 用户角色

| 角色 | 权限级别 | 描述 |
|------|---------|------|
| **Admin (管理员)** | 最高 | 家庭创建者，拥有所有权限，可管理成员和设置 |
| **Editor (编辑者)** | 中等 | 可添加/编辑/删除记录，查看所有数据，不能管理成员 |
| **Viewer (查看者)** | 最低 | 仅可查看记录，不能添加或修改任何数据 |

## 功能需求

### 场景1：家庭管理员邀请新成员

#### FR-1.1: 邀请码生成与分享
**用户故事：** 作为家庭管理员，我想要生成邀请码并分享给家人，以便他们加入家庭共同记录宝宝成长

**验收标准：**
1. When 管理员点击"邀请家人"，the system shall 显示当前邀请码和有效期倒计时
2. When 邀请码即将过期（剩余<24小时），the system shall 显示警告提示
3. When 管理员点击"重新生成"，the system shall 生成新的6位邀请码并使旧码失效
4. When 管理员点击"分享"，the system shall 生成包含邀请码的小程序分享卡片
5. If 邀请码已过期，then the system shall 自动重新生成新码

#### FR-1.2: 邀请链接分享（可选增强）
**用户故事：** 作为家庭管理员，我想要通过微信直接分享邀请链接，让家人一键加入

**验收标准：**
1. When 管理员选择"微信邀请"，the system shall 生成带参数的小程序路径
2. When 被邀请人点击分享卡片，the system shall 自动填充邀请码并跳转加入流程

### 场景2：新成员加入家庭

#### FR-2.1: 邀请码输入与验证
**用户故事：** 作为被邀请的家庭成员，我想要输入邀请码加入家庭，以便共同查看和记录宝宝数据

**验收标准：**
1. When 用户输入6位邀请码，the system shall 实时验证格式有效性
2. When 用户提交邀请码，the system shall 在2秒内返回验证结果
3. If 邀请码无效或已过期，then the system shall 显示"邀请码无效或已过期，请联系管理员"
4. If 用户已是该家庭成员，then the system shall 提示"您已是家庭成员，无需重复加入"
5. When 验证通过，the system shall 显示家庭名称和成员数量，请求确认加入

#### FR-2.2: 加入后数据同步
**用户故事：** 作为新加入的成员，我想要立即看到家庭的宝宝和记录数据

**验收标准：**
1. When 用户成功加入家庭，the system shall 自动加载家庭的宝宝列表
2. When 有多个宝宝，the system shall 默认选中第一个宝宝并显示其记录
3. When 加入成功，the system shall 在首页显示"成功加入XX家庭"的提示
4. When 新成员首次查看记录，the system shall 显示"与XX位家人共同记录"的提示

---

### 场景3：家庭管理员管理成员权限

#### FR-3.1: 成员列表与权限展示
**用户故事：** 作为家庭管理员，我想要查看所有成员及其权限，以便了解谁可以做什么

**验收标准：**
1. When 管理员进入家庭管理页，the system shall 显示所有成员列表（包括头像、昵称、角色、加入时间）
2. When 显示成员角色，the system shall 使用中文标签："管理员"/"成员"/"仅查看"
3. When 管理员查看自己，the system shall 标注"我"和"创建者"标识
4. When 列表中有多个成员，the system shall 按加入时间倒序排列

#### FR-3.2: 成员权限修改
**用户故事：** 作为家庭管理员，我想要修改成员的权限，以便控制他们对数据的访问级别

**验收标准：**
1. When 管理员点击成员卡片，the system shall 弹出权限选择菜单
2. When 选择新权限，the system shall 显示确认对话框说明权限变更影响
3. When 确认变更，the system shall 在1秒内更新权限并显示成功提示
4. If 尝试将唯一管理员降级，then the system shall 拒绝并提示"家庭至少需要一位管理员"
5. When 权限变更成功，the system shall 发送订阅消息通知被修改成员（如用户订阅）

#### FR-3.3: 移除家庭成员
**用户故事：** 作为家庭管理员，我想要移除不再参与记录的家庭成员，以保护数据安全

**验收标准：**
1. When 管理员左滑成员卡片或点击"移除"，the system shall 显示二次确认对话框
2. When 确认移除，the system shall 要求输入"确认移除"文字防止误操作
3. When 成员被移除，the system shall 立即撤销其对家庭数据的访问权限
4. When 成员被移除，the system shall 保留该成员创建的历史记录（显示为"已退出成员"）
5. When 移除成功，the system shall 发送通知给被移除成员（如用户订阅）

---

### 场景4：普通成员查看和操作记录

#### FR-4.1: 记录创建者标识
**用户故事：** 作为家庭成员，我想要知道每条记录是谁添加的，以便了解宝宝的照顾情况

**验收标准：**
1. When 显示记录列表，the system shall 在每条记录上显示创建者头像和昵称
2. When 创建者已退出家庭，the system shall 显示"已退出成员"占位标识
3. When 记录是"我"创建的，the system shall 可选择显示"我"而非昵称
4. When 查看记录详情，the system shall 显示完整的创建者信息和创建时间

#### FR-4.2: 基于角色的操作权限
**用户故事：** 作为不同角色的成员，我想要根据权限执行相应的操作

**验收标准：**
1. While 用户角色为Viewer，the system shall 隐藏所有"添加记录"按钮
2. While 用户角色为Viewer，the system shall 禁用编辑和删除操作，显示"仅查看权限"提示
3. When Editor尝试删除他人创建的记录，the system shall 显示确认对话框"确定删除XX创建的记录？"
4. When Admin删除任何记录，the system shall 直接执行无需额外确认
5. When 用户无权限执行操作，the system shall 显示友好的权限不足提示

#### FR-4.3: 实时数据同步
**用户故事：** 作为家庭成员，我想要实时看到其他成员添加的记录

**验收标准：**
1. When 其他成员添加记录，the system shall 在首页时间线实时显示（轮询或推送）
2. When 检测到新记录，the system shall 显示"XX刚刚添加了记录"的轻提示
3. When 用户处于记录列表页，the system shall 自动刷新显示新记录
4. When 网络离线时，the system shall 在恢复连接后自动同步数据

---

### 场景5：成员退出家庭

#### FR-5.1: 主动退出家庭
**用户故事：** 作为家庭成员，我想要主动退出家庭，以便在不再参与照顾时保护隐私

**验收标准：**
1. When 普通成员点击"退出家庭"，the system shall 显示确认对话框说明退出后果
2. When 确认退出，the system shall 在1秒内完成退出并清理本地家庭数据
3. When 成员退出，the system shall 保留其创建的历史记录（显示为"已退出成员"）
4. When 退出成功，the system shall 跳转至引导页，显示"您已退出家庭"

#### FR-5.2: 管理员退出处理
**用户故事：** 作为家庭管理员，我想要在退出前妥善处理管理权限

**验收标准：**
1. If 管理员是家庭唯一成员，then the system shall 提示"退出将解散家庭"并提供确认
2. If 家庭还有其他成员，then the system shall 要求先指定新管理员或移除所有成员
3. When 管理员确认解散家庭，the system shall 删除家庭记录并解除所有成员关联
4. When 家庭解散，the system shall 保留宝宝的记录数据（关联到原创建者个人）

#### FR-5.3: 被移除成员的退出
**用户故事：** 作为被管理员移除的成员，我想要知道已被移除并清理数据

**验收标准：**
1. When 成员被移除后打开小程序，the system shall 检测到成员身份失效
2. When 检测到身份失效，the system shall 显示"您已不在该家庭中"并引导重新加入或创建家庭
3. When 被移除成员确认，the system shall 清理本地家庭相关数据
4. When 清理完成，the system shall 跳转至引导页面

## 非功能需求

### NFR-1: 性能要求
- 邀请码验证响应时间 < 2秒
- 成员列表加载时间 < 1秒（10个成员以内）
- 权限变更实时生效，无需重新登录
- 数据同步延迟 < 5秒（正常网络条件下）

### NFR-2: 安全要求
- 邀请码有效期7天，过期自动失效
- 邀请码6位随机生成，避免可预测性
- 成员移除后立即撤销数据访问权限
- 所有权限操作记录操作日志（可选）

### NFR-3: 用户体验
- 操作确认对话框防止误操作
- 权限变更时清晰说明影响范围
- 离线状态下显示"同步中"状态指示
- 新成员加入时有友好的引导提示

### NFR-4: 兼容性
- 支持现有家庭数据平滑迁移
- 旧版本小程序能正常使用基础功能
- 角色字段缺失时默认为Editor

## 边界条件和异常处理

| 场景 | 处理方式 |
|------|---------|
| 邀请码被暴力破解尝试 | 限制单个邀请码验证次数（5次/小时） |
| 网络中断时加入家庭 | 本地缓存邀请码，恢复后自动重试 |
| 管理员账号异常（注销） | 自动将最老的Editor提升为Admin |
| 同时多人修改同一成员权限 | 使用乐观锁，后提交者收到冲突提示 |
| 成员退出时正在编辑记录 | 保存草稿，下次加入时恢复 |
| 家庭数据量过大（>1000条记录） | 分页加载，显示加载进度 |
